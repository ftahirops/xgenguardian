# fp-bench — accuracy measurement harness

Measures false-positive (FP) and true-positive (TP) rates of the XGG verdict
engine against labeled URL corpora. **Every detection change must pass this
gate before merging.**

## Why this exists

Without continuous measurement, every new detection feature is guesswork —
you can't tell if it helped recall or hurt precision. This harness lets you:

- See FP rate per release on a stable benign corpus
- See TP rate on fresh malicious feeds
- Identify which reason codes drive FPs
- Block merges that regress accuracy

## Quick start

```bash
cd tools/fp-bench

# 1. Fetch corpora (~1 minute, do once per day)
make fetch

# 2. Run the engine against everything (~5-15 minutes depending on size)
make bench

# 3. Pass/fail gate
make gate

# 4. View report
make report
```

## Targets

| `make <target>` | What |
|---|---|
| `fetch` | Refresh Tranco + URLhaus + OpenPhish into `corpus/` |
| `bench` | Full benchmark, writes `reports/bench-<ts>.{json,md}` |
| `smoke` | 50-URL/label quick test for fast iteration |
| `gate`  | Pass/fail against thresholds, exits non-zero on regression |
| `report`| Print latest markdown report |
| `all`   | fetch + bench + gate |

## Thresholds (override via env)

| Env var | Default | Meaning |
|---|---|---|
| `MAX_FP_RATE` | `0.001` | 0.1% FP cap on benign |
| `MIN_TP_RATE` | `0.85`  | 85% TP floor on malicious |
| `MAX_BENIGN_HITS_PER_CODE` | `5` | High-confidence codes may not fire >N times on benign |
| `MIN_BENIGN_SAMPLE` | `100` | Benign sample must be ≥ N URLs |

## Corpora

- `corpus/benign-curated.txt` — committed, hand-maintained baseline (banks, SaaS, our infra)
- `corpus/benign-tranco.txt`  — generated by `fetch`, Tranco top-N
- `corpus/malicious-urlhaus.txt` — generated, last-7-days URLhaus
- `corpus/malicious-openphish.txt` — generated, current OpenPhish feed

## Wiring into CI

Add to your CI workflow:
```yaml
- run: cd tools/fp-bench && make fetch bench gate
```

The gate exits 1 if FP rate regresses past target, blocking the merge.

## Reading reports

`reports/bench-latest.md` is the human-readable view. Key sections:

1. **Benign corpus** — FP rate + table of every URL that was incorrectly flagged
2. **Malicious corpus** — TP rate + sample of misses
3. **Reason-code signal quality** — for each code, ratio of malicious-hits
   to (benign-hits + malicious-hits). Codes < 0.7 are noisy and need tuning.
4. **Latency** — p50/p95/p99 per-URL classification time

## Interpreting reason-code signals

If a code shows up high on benign URLs, that's where to invest tuning:

| Code | Benign hits | Malicious hits | Signal | Action |
|---|---|---|---|---|
| `DGA_SUSPECT` | 12 | 8 | 0.40 | 🔴 Inverted — disable or rescore |
| `RDAP_DOMAIN_NEW` | 22 | 47 | 0.68 | ⚠️ Too noisy alone, must combine |
| `VISUAL_REPLICA_HIGH` | 0 | 35 | 1.00 | ✅ Clean signal |

## When to refresh corpus

- `make fetch` daily in CI
- Hand-add to `benign-curated.txt` whenever you find a FP not covered
- Hand-add new known-bad URLs to a `corpus/malicious-curated.txt` as you
  encounter ones the feeds miss
